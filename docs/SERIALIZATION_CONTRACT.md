# axjs ↔ Ax/BoTorch Serialization Contract

**Status:** Proposal
**Audience:** Ax team, BoTorch team

## What axjs does

axjs is a TypeScript library that replicates BoTorch GP posterior predictions
(`model.posterior(X).mean` and `.variance`) in the browser. It enables client-side
visualization of GP response surfaces, slice plots, and model diagnostics without
a Python backend.

It is **prediction-only** — no fitting. A Python export step serializes a fitted
BoTorch model to JSON, and the TypeScript side deserializes and predicts.

## The problem

There is no public BoTorch API for "give me everything needed to replicate
predictions." The export code (`_extraction.py`) crawls the model object tree
using a mix of public, semi-public, and private attributes. This broke when
BoTorch 0.16 changed `IndexKernel` → `PositiveIndexKernel` and `ConstantMean` →
`MultitaskMean`. It will break again.

Additionally, Ax adapter-level transforms (LogY, BilogY, etc.) are applied
*before* data reaches BoTorch and are invisible in the model state. Without
separate metadata, client-side predictions are in the wrong space.

## Current parity evidence

21 fixtures verify numerical parity at Float64 precision against BoTorch 0.17.2:

| Model type | Fixtures | Worst discrepancy |
|---|---|---|
| SingleTaskGP (Matern/RBF, ARD) | 4 | ~1e-16 |
| FixedNoiseGP (homo/heteroscedastic) | 2 | ~1e-15 |
| Input transforms (Normalize, Warp) | 2 | ~1e-15 |
| Composite kernels (Product, Additive) | 2 | ~1e-15 |
| ModelListGP (multi-output) | 3 | ~1e-15 |
| MultiTaskGP (ICM, +categorical, +warp) | 3 | ~1e-11 |
| PairwiseGP (BOPE, +warp) | 2 | ~1e-13 |
| EnsembleGP / SAAS (NUTS + MAP) | 2 | ~1e-14 |
| Integer parameters | 1 | ~1e-16 |

All discrepancies are 5+ orders of magnitude below the 1e-5 tolerance ceiling,
which is enforced in CI. These fixtures are generated by fitting real BoTorch
models on benchmark functions (Branin, Hartmann, BraninCurrin), capturing the
fitted state, and comparing predictions.

## What we need from BoTorch/GPyTorch

### The attributes we access today

The extraction code currently reaches into these model internals:

**Private (high breakage risk):**
| Attribute | What we need it for |
|---|---|
| `model._task_feature` | Which column of train_X is the task index |
| `task_kernel._eval_covar_matrix()` | The fully-evaluated B matrix from PositiveIndexKernel |
| `kernel.raw_var` | Legacy IndexKernel diagonal variance (pre-0.16) |
| `mean.raw_constant` | Fallback for MultitaskMean constants |

**Semi-public (stable so far):**
| Attribute | What we need it for |
|---|---|
| `model.train_inputs[0]`, `model.train_targets` | Training data |
| `model.covar_module` | Kernel tree to walk recursively |
| `model.mean_module.constant` | Mean function constant(s) |
| `model.likelihood.noise` | Noise variance |
| `model.input_transform` | Normalize / Warp / ChainedInputTransform |
| `model.outcome_transform` | Standardize (means, stdvs) |
| `kernel.lengthscale`, `kernel.outputscale` | Kernel hyperparameters |
| `kernel.active_dims` | Dimension selection |
| `kernel.base_kernel`, `kernel.kernels` | Kernel tree structure |
| `kernel.covar_factor` | IndexKernel/PositiveIndexKernel W matrix |

### Ideal: a `prediction_state()` method

The cleanest fix would be a BoTorch method that returns a self-contained dict
with everything needed for prediction:

```python
state = model.prediction_state()
# {
#   "model_type": "SingleTaskGP",
#   "train_X": Tensor,
#   "train_Y": Tensor,
#   "kernel": { "type": "RBF", "lengthscale": [...], ... },
#   "mean_constant": float | list[float],
#   "noise_variance": float | list[float],
#   "input_transform": { ... } | None,
#   "outcome_transform": { ... } | None,
#   # MultiTaskGP-specific:
#   "task_feature": int,
#   "num_tasks": int,
#   "task_covar": { "covar_matrix": [[...]] },
# }
```

This would:
- Give BoTorch ownership of which internals matter for prediction
- Let BoTorch add a regression test: `predict_from_dict(model.prediction_state(), X) ≈ model.posterior(X)`
- Eliminate private attribute access by axjs
- Be versionable (include a `format_version` field)

### Minimal alternative: stabilize 5 attributes

If `prediction_state()` is too large a surface area, we'd benefit from just
making these 5 attributes public/stable:

1. **`MultiTaskGP.task_feature`** → currently `_task_feature` (private)
2. **`PositiveIndexKernel.covar_matrix`** → property returning `_eval_covar_matrix()` result
3. **`MultitaskMean.constants`** → list of per-task mean constants
4. **`ModelListGP.outcome_names`** → currently `_outcome_names` (private)
5. **`IndexKernel.var`** → public access to diagonal variance (not `raw_var`)

## What we need from Ax

### Adapter-level transforms are invisible to the model

When Ax applies transforms like `LogY` at the adapter layer:

```
Raw Y_raw → adapter applies log(Y_raw) → BoTorch sees log-space Y
→ BoTorch trains GP → predictions in log-space → adapter applies exp()
→ user sees original-space predictions
```

If we export the BoTorch model directly, predictions are in **log-space**.
The adapter transforms are stored on the `Adapter` instance
(`adapter.transforms: OrderedDict[str, Transform]`), not on the model.

### What we need exported

When `export_client()` or `export_experiment()` is called, we need the
adapter to also emit which Y-transforms were applied:

```python
# From the fitted Adapter (GenerationStrategy.model):
adapter = generation_strategy.model  # TorchAdapter
y_transforms = []
for name, tf in adapter.transforms.items():
    if name in ("LogY", "BilogY", "PowerTransformY", "StandardizeY"):
        info = {"type": name}
        # LogY: which metrics were log-transformed
        if hasattr(tf, "trial_indices"):
            info["metrics"] = list(tf.metric_names)
        # PowerTransformY: fitted lambda parameters
        if hasattr(tf, "power_transforms"):
            info["power_params"] = {
                k: v.lambdas_.tolist()
                for k, v in tf.power_transforms.items()
            }
        y_transforms.append(info)
```

### Ideal: `Adapter.prediction_metadata()`

A method that returns all metadata needed for client-side un-transformation:

```python
metadata = adapter.prediction_metadata()
# {
#   "y_transforms": [
#     {"type": "LogY", "metrics": ["latency"]},
#     {"type": "StandardizeY", "Ymean": {...}, "Ystd": {...}},
#   ],
#   "x_transforms": [
#     {"type": "IntToFloat", "parameters": ["batch_size"]},
#   ],
# }
```

### Minimal alternative: document the introspection path

If a method is too much, just document that `adapter.transforms` is the
stable way to introspect which transforms were applied. Currently this is
used implicitly but never documented as a public API.

## The JSON schema we use (format_version 2)

This is the contract between Python export and TypeScript consumption.
axjs owns this schema. BoTorch/Ax don't need to adopt it — we just need
stable access to the underlying data.

### Top-level structure
```json
{
  "format_version": 2,
  "botorch_version": "0.17.2",
  "ax_version": "1.2.4",
  "search_space": {
    "parameters": [
      { "name": "x0", "type": "range", "bounds": [0.0, 1.0] },
      { "name": "x1", "type": "choice", "values": [0, 1, 2] }
    ]
  },
  "model_state": { ... },
  "adapter_transforms": [ ... ]
}
```

### Kernel state (recursive tree)
```
Base:       { "type": "Matern"|"RBF"|"Categorical", "lengthscale": [...], "nu": 2.5, "active_dims": [...] }
Scaled:     { "type": "Matern", ..., "outputscale": 0.5 }     // legacy flat format
Scale+Comp: { "type": "Scale", "outputscale": 0.5, "base_kernel": { "type": "Product", "kernels": [...] } }
Composite:  { "type": "Additive"|"Product", "kernels": [...] }
```

### Outcome transforms
```
Standardize: { "type": "Standardize", "mean": 0.5, "std": 1.2 }
Log:         { "type": "Log" }
Bilog:       { "type": "Bilog" }
Power:       { "type": "Power", "power": 0.5 }
Chained:     { "type": "Chained", "transforms": [ ... ] }
```

### MultiTaskGP task covariance
```
New (0.16+):  { "covar_factor": [[...]], "covar_matrix": [[...]] }    // pre-computed B
Legacy:       { "covar_factor": [[...]], "log_var": [...] }            // B = WW^T + diag(exp(v))
```

## Transform pipeline diagram

```
                        Ax Adapter                              BoTorch Model
                   ┌─────────────────┐                  ┌────────────────────────┐
                   │                 │                  │                        │
   Raw Y_raw ──────▶  LogY / BilogY  ├──── Y_adapted ──▶  Standardize (model)   │
                   │  PowerTransformY│                  │  Log (model)           │
                   │  StandardizeY   │                  │  outcome_transform     │
                   │                 │                  │                        │
                   │                 │                  │                        │
   Raw X_raw ──────▶  IntToFloat     ├──── X_adapted ──▶  Normalize              │
                   │  OneHot         │                  │  Warp (Kumaraswamy)    │
                   │  Normalize      │                  │  input_transform       │
                   │                 │                  │                        │
                   └─────────────────┘                  └────────────────────────┘
                                                                    │
                   INVISIBLE to model export                        │ VISIBLE
                   (need adapter metadata)               ┌──────────┴──────────┐
                                                         │  axjs JSON export   │
                                                         │  (prediction_state) │
                                                         └─────────────────────┘
```

The critical gap: adapter-level transforms are on the **left side** of this
diagram. If the user configured `LogY`, axjs currently has no way to know that
predictions should be exponentiated.

## Summary of asks

| Ask | Team | Effort | Impact |
|---|---|---|---|
| `model.prediction_state()` | BoTorch | Medium | Eliminates all private attr access |
| *or* stabilize 5 attributes | BoTorch | Small | Covers highest-risk breakage points |
| `adapter.prediction_metadata()` | Ax | Small | Enables adapter transform round-trip |
| *or* document `adapter.transforms` as public | Ax | Tiny | At least gives us a stable path |
